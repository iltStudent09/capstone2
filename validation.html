Customer Manager Validation and Business Rules

Reference document for implementing consistent validation in add/edit customer flows.

1) Customer Identity Rules

  • id is generated by the backend and must not be entered by users on create.
  • id is immutable after creation.
  • email should be unique across all customers.

2) Required Fields

  • Required: name, email, phone, address, city, state, zip.
  • All text fields should be trimmed before validation and save.
  • state must be exactly 2 uppercase letters (example: TX).
  • zip must match selected format rule (example: 5 digits, optional 5+4 if enabled).
  • email must be a valid email format.

3) Add Customer Rules

  • Reject create if any required field is missing or invalid.
  • Reject create if email already exists.
  • Disable submit while save request is in progress.
  • On success, navigate to customer list.

4) Edit Customer Rules

  • Edit route must include a valid existing customer id.
  • Prefill form from backend record before edit.
  • Reject update if required fields are invalid.
  • Keep id unchanged in update payload logic.
  • On success, navigate to customer list.

5) API and Error Handling Rules

  • Show clear error message when fetch/save fails.
  • Keep client-side and server-side validation rules aligned to prevent validation drift.
  • Use one shared validation source so add/edit enforce identical rules.

6) Suggested Validation Functions (for later implementation)

  • normalizeCustomerInput(formData) for trim and uppercase normalization.
  • validateCustomer(formData) returning field-level errors and form validity.
  • isEmailUnique(email, currentId?) for add and edit duplicate checks.
This file is intended as implementation guidance. You can later map each rule into shared schema validation and reducer actions.
